GOAL                                            *nvim-remote-mirror-goal*

Build a local-first reusable remote workspace daemon, with Neovim as the only
client for now, that keeps cached navigation, opening, status, provisional
search, and editing responsive over slow, unstable, or offline SSH links by
operating from a durable checksum-verified local mirror. Hydrate missing and
related content in bounded batches, run source-adjacent tools such as LSP near
the remote tree, preserve saves through snapshots and checksum compare-and-swap
retries, and keep SSH behind a transport-neutral sidecar/agent boundary.
By default the Neovim facade launches one stdio sidecar process per editor
session. For a reusable endpoint, set connection = "socket"; Neovim will connect
to a Unix-domain sidecar listener or start one with the same JSON
command/notification protocol and durable workspace state.

INSTALL                                      *nvim-remote-mirror-install*

Requirements:
    - Neovim 0.10+, with Neovim 0.11+ preferred.
    - Rust toolchain on the local machine.
    - SSH access for POSIX ssh://host/path or Windows ssh://host/B:/path targets.
    - nrm-agent installed and executable on each remote host, installable from
      a trusted signed registry, or uploadable with :RemoteInstallAgent.

Build the binaries from the checkout:
>
  cargo build --release
<

For example, install nrm-agent manually on a POSIX host:
>
  scp target/release/nrm-agent myhost:~/.local/bin/nrm-agent
  ssh myhost 'chmod +x ~/.local/bin/nrm-agent'
<

Or connect and run the explicit repair flow:
>
  :RemoteConnect ssh://myhost/home/me/project
  :RemoteHealth
  :RemoteInstallAgent
<
No registry is configured by default, so default connect does not install or
update remote binaries. With SSH, a configured trusted signed registry, and
remote_agent_auto_install=true, connect installs or repairs known agent
failures from a verified Linux, macOS, or Windows x64/ARM64 artifact. It never
falls back to an unsigned local binary. Set remote_agent_auto_install=false to
keep registry-backed repair explicit. Explicit install/update remains
available; without a registry it uploads the configured local agent.

Windows OpenSSH targets use an absolute drive root and forward slashes:
>
  :RemoteConnect ssh://windows-host/B:/repos/project
<
UNC paths, drive-relative paths, and backslash target syntax are unsupported.

Non-interactive SSH may not load ~/.profile or shell rc files. If this fails:
>
  ssh myhost 'command -v nrm-agent'
<
set remote_agent to an absolute remote path such as
/home/me/.local/bin/nrm-agent.

Minimal lazy.nvim checkout config:
>
  {
    dir = vim.fn.expand("~/repos/nvim-remote-mirror"),
    name = "nvim-remote-mirror",
    config = function()
      require("nvim_remote_mirror").setup({
        sidecar = vim.fn.expand("~/repos/nvim-remote-mirror/target/release/nrm-sidecar"),
        agent = vim.fn.expand("~/repos/nvim-remote-mirror/target/release/nrm-agent"),
        remote_agent = "/home/me/.local/bin/nrm-agent",
        connection = "socket",
      })
    end,
  }
<

UI                                                *nvim-remote-mirror-ui*

:RemoteWorkspace
    Open the remote workspace dashboard. The dashboard shows connection state,
    remote health, mirror root, files root, mirror counts, background scan
    state, save queue counts, automatic-bootstrap result/reason/error, and
    signed-registry platform, source, cache, digest, signing-key, and stable
    error-code diagnostics.

:RemoteConnectUI
    Prompt for a local path, POSIX ssh://host/absolute/path, or Windows
    ssh://host/B:/absolute/path target and connect.

:RemoteFiles
    Prompt for a path query, show known mirror paths with vim.ui.select(), and
    open the selected remote path.

:RemoteGrepUI
    Prompt for grep text and run :RemoteGrep.

:RemoteQueue
    Show queued saves with actions to open local copies, snapshots, conflict
    copies, diff conflicts, accept local or remote conflict content, retry
    queued saves, or refresh the queue.

:RemoteConflicts
    Show only conflict entries from the save queue. The query is filtered by
    the sidecar, so large pending queues do not hide later conflict entries.

Dashboard keys:
    c connect        h health         o open path      f find files     g grep
    z cwd            s save queue     C conflicts      F flush queue
    r refresh        d disconnect     R reconnect      q/x close

COMMANDS                                      *nvim-remote-mirror-commands*

:RemoteConnect [target]
    Connect to a local path, POSIX ssh://host/absolute/path, or Windows
    ssh://host/B:/absolute/path target. Startup uses
    the durable local mirror first; the remote agent handshake is lazy, so
    cached opens, cached grep, and status can work while SSH is unreachable.
    Failed remote attempts use a capped adaptive lane backoff before trying SSH
    again. When signed automatic install is enabled, connect repairs only a
    missing, non-executable, version-mismatched, or protocol-mismatched agent.
    A missing root, unavailable host, or unknown failure is not mutated. A
    bootstrap failure leaves the local-first connection available in a
    degraded state.

:RemoteReconnect
    Reconnect to the last remote target, reusing the durable mirror and
    retrying queued saves after a remote probe succeeds. Remote buffers saved
    while disconnected are also replayed into the next sidecar session so their
    local mirror files can be snapshotted. If Neovim cannot write to a closed
    sidecar channel, the request fails immediately and reconnect handling starts
    without waiting for request_timeout_ms.

:RemoteCd
    Set the current tab's working directory to the active workspace's local
    mirror files root. This is useful for cwd-based local plugins that should
    operate on the mirrored project tree.

:RemoteUseLocal
    Clear the current tab's remote runtime binding. This does not disconnect,
    restore the cwd changed by :RemoteCd, or relabel a remote-owned buffer.
    Plain buffers then select the local runtime; a remote-owned buffer still
    selects NRM unless a Lua caller requests authority="local" explicitly.

:RemoteTrustWorkspace[!]
    Persist process/terminal trust for the resolved workspace. Without [!],
    confirm first; [!] is an intentional scripted opt-in that skips the
    confirmation. Connect never grants this trust or starts an arbitrary
    workspace command.

:RemoteUntrustWorkspace
    Remove persisted process/terminal trust for the resolved workspace.

:RemoteTerminal[!] [cmd...]
    Prepare terminal readiness and open an attached remote PTY in a bottom
    split. With no cmd, start the authority account's default shell
    and enter terminal-input mode; otherwise pass cmd as an argv list, never
    shell-interpolated text. A later prepare or launch failure is reported by
    notification. The local bridge disables its own echo and control-key
    processing so only the authority PTY handles them. [!] requests detached
    persistence and currently fails with persistence_unavailable until the
    persistent runtime broker is implemented.

:RemoteOpen {path}
    Open a workspace-relative file from the local mirror when cached, including
    dirty, stale, or deleted mirror entries. Hydrate it from the remote when no
    cached file is available. Large uncached files hydrate in fixed-size chunks
    and verify the full remote hash on the final chunk. When configured, after
    opening it can prefetch a small batch of related files from existing mirror
    metadata.

:RemoteOpen! {path}
    Force remote rehydration for clean cached files. Dirty cached files are not
    overwritten; if the local file is missing, the latest queued save snapshot
    is restored instead.

:RemoteScan [limit]
    Scan remote metadata into the local mirror index. This populates known
    paths for navigation without hydrating every file. Omit limit to use the
    sidecar default.

:RemoteFind [query]
    Search known mirror paths locally and fill quickfix with mirror-local
    filenames. Selecting an uncached result hydrates it lazily through the
    mirror buffer autohydration path.

:RemoteGrep {query}
    Queue remote grep, search a bounded slice of already hydrated local mirror
    files as a provisional cache result, walk the remote tree in session-backed
    pages, hydrate remote matches in a batch, and refresh quickfix with safe
    local mirror paths. Cached grep uses a persisted local line/trigram index
    and keeps dirty cached matches as local truth.

:RemotePrefetch {path...}
    Hydrate one or more workspace-relative files in the background request.

:RemoteMirrorStart
    Start the background mirror builder. It probes the remote, scans metadata
    with a cursor, then hydrates a small batch of clean uncached known files.
    The scan cursor is persisted in the mirror database, so reconnects resume
    metadata discovery from the last completed background batch.

:RemoteMirrorStop
    Stop the background mirror builder.

:RemoteFlush
    Flush the current remote buffer to the remote workspace using checksum
    compare-and-swap. Saves above 4 MiB are uploaded in 1 MiB chunks. If no
    sidecar is connected, the path is marked for replay after reconnect. Buffers
    are tagged with the workspace that opened them; if another target is active,
    the save is deferred until the original workspace is reconnected.
    The remote agent uses a cross-platform cooperative write lock keyed by the
    canonical target for nrm writers, then rechecks the remote base hash before
    replacement. Windows uses native replacement APIs for existing-file CAS
    writes. Non-cooperating remote writers can still race at the filesystem
    boundary.
    Automatic BufWritePost flushing uses the actual write target, but untracked
    mirror-root paths are skipped by default. Use :RemoteAdopt for intentional
    new remote files, or set adoption_policy="auto" for legacy auto-adoption.

:RemoteAdopt [path]
    Explicitly create or take over a mirror-root path as a workspace-relative
    remote file. Use this for intentional new remote files or recreating a file
    that validation marked deleted. If the sidecar is connected but the remote
    is unavailable, the sidecar snapshots the bytes into the durable queue.

:RemoteFlushQueue
    Retry queued or previously failed remote saves and report applied, still
    queued, conflict, and remaining counts.

:RemoteAcceptLocalConflict {queue_id}
    Resolve the latest conflict for a path by retrying its durable local save
    snapshot against the recorded remote conflict hash. Older unresolved saves
    for that path are superseded. The command refuses stale queue rows when
    newer unresolved saves exist.

:RemoteAcceptRemoteConflict {queue_id}
    Resolve the latest conflict for a path by replacing the local mirror file
    with the full saved remote conflict copy and marking older unresolved saves
    for that path resolved. The command refuses partial remote conflict copies
    and stale queue rows when newer unresolved saves exist.

:RemoteSaveQueue [limit]
    List pending, failed, unreplayable, and conflicted saves from the local
    durable queue in quickfix. This command reads local mirror state only, so it
    remains useful while the remote is unavailable.

:RemoteValidate [path]
    Validate the current remote buffer or a workspace-relative path against the
    remote checksum.

:RemoteRefresh [path...]
    Validate multiple cached files in one remote request and mark mirror entries
    valid, stale, or deleted.

:RemoteStatus
    Show known, cached, indexed, dirty, and pending mirror counts plus the
    plugin connection state, pending reconnect or explicit disconnect state,
    background metadata scan state, LSP summary, and last known remote health.
    This is a local fast path and does not probe SSH.

:RemoteHealth
    Probe the remote agent explicitly and update last-known remote health. The
    result includes agent compatibility state such as ok, missing_agent,
    version_mismatch, protocol_mismatch, agent_not_executable, or
    remote_root_missing when the sidecar can classify the failure.

:RemoteInstallAgent[!] [path]
    Transactionally install an agent on an SSH target. Registry mode downloads
    and verifies the matching native build locally; otherwise the command uses
    the configured local agent bytes. Without [path], a safe bare remote_agent
    is installed under $HOME/.local/bin on POSIX or %LOCALAPPDATA%\nrm\bin on
    Windows, preserving its command name and adding .exe on Windows when
    needed. [!] permits replacing an existing target. The candidate passes
    exact --version and staged Hello checks before activation, then a
    normal-path Hello; failures restore and reprobe the previous executable.

:RemoteUpdateAgent[!] [path]
    Use the same transactional flow when :RemoteHealth reports a missing,
    incompatible, or non-executable remote agent. Without [!], a compatible
    agent is skipped. Registry mode fails closed and never falls back to the
    configured local agent.

:RemoteGitStatus [path...]
    Run remote `git status --porcelain=v1 -z --branch --untracked-files=all`
    through the agent and put changed paths in quickfix. Optional paths are
    workspace-relative or mirror-local paths.

:RemoteGitDiff [path]
    Run remote `git diff` for the current remote buffer or given path and open
    the result in a nofile diff scratch buffer.

:RemoteGitBlame [path]
    Run remote `git blame` for the current remote buffer or given path and put
    blame lines in quickfix.

:RemoteLspStart {cmd...}
    Start a simple remote LSP command for the active workspace. For complex
    server configs, prefer start_lsp() from Lua.

:RemoteLspStop
    Stop active remote LSP clients for the current workspace.

:RemoteLspRestart [cmd...]
    Restart the last remote LSP command, or restart with a new simple command
    when arguments are provided.

:RemoteLspStatus
    Show active remote LSP clients, last command, last error, workspace, and
    last-known remote health.

:RemoteDisconnect
    Stop the sidecar and agent.

Each setup() call is a complete configuration snapshot over the built-in
defaults. Omitted options reset to their defaults instead of retaining values
from an earlier call. In particular, omitting remote_agent_registry_url
disables registry mode, and remote_agent_registry_public_keys replaces the
previous key table rather than merging with it. Pass the full desired
configuration on every call.

For ssh:// targets, remote_agent is executed on the remote host. With registry
mode disabled, the local agent path is the explicit install/update source. With
registry mode enabled, only the matching signed artifact is used; the local
agent is not a fallback. remote_agent_auto_install defaults to true but is
inactive unless both an SSH target and a trusted signed registry are configured.
In that opt-in mode connect skips a compatible agent, installs a missing agent,
and repairs agent_not_executable, version_mismatch, or protocol_mismatch. It
does not mutate remote_root_missing, unavailable, or unclassified states. Set
remote_agent_auto_install=false to keep registry operations explicit. A failed
automatic bootstrap leaves the mirror connected in a degraded state and is
reported separately in :RemoteWorkspace.
Workspace runtime prepare() never invokes this repair path. It performs a
read-only authority probe and may persist only an explicit local trust
decision. Missing or incompatible agents fail preparation until connect-time
signed repair or an explicit :RemoteInstallAgent/:RemoteUpdateAgent succeeds.
On POSIX, a safe bare remote_agent installs under $HOME/.local/bin with the
same command name and SSH launch prepends that directory to PATH. On Windows
it installs under %LOCALAPPDATA%\nrm\bin and gains an .exe suffix when needed.
Relative agent paths containing a slash are rejected for default installation.
An absolute remote_agent, remote_agent_install_path, or optional command path
overrides the default.
Non-interactive SSH may not load the same PATH as an interactive login shell.
If :RemoteHealth reports missing_agent, run :RemoteInstallAgent or set
remote_agent to an absolute remote path.
Local-binary install copies the configured bytes as-is and does not
cross-compile. Registry mode detects Linux, macOS, or Windows x64/ARM64 and
selects one of six fixed native release targets.
Set connection = "socket" to connect through a reusable Unix-domain sidecar
listener. If socket_path is unset, the plugin derives a stable path under
socket_dir, state_dir/sockets, or the system temp directory from the target and
daemon-affecting config. The listener is detached from Neovim and currently
serves one active client session at a time; :RemoteDisconnect closes the client
session but leaves the listener available for the next editor. The sidecar
reports this as workspace_info.client_mode = "single_writer" with a
workspace_info.client_policy object so future clients can distinguish this
sequential policy from future multi-client modes.
The directory immediately containing the socket must be owned by the current
user with mode 0700. Existing sockets must be owned by the current user, must
not be symlinks, and may have no permissions beyond 0600. Lexical and resolved
ancestors must be owned by the current user or root and cannot be group/world-
writable unless a sticky ancestor protects the next entry. A private leaf
below sticky /tmp is supported; a socket directly in /tmp is rejected.
Automatic agent bootstrap requires that derived socket identity. An explicit
socket_path leaves connect local-first but skips automatic mutation; manage the
fixed-path daemon and use an explicit install/update command instead.
When state_dir is unset, durable mirror state uses the platform state directory,
typically ~/.local/state/nvim-remote-mirror on Linux. Use :RemoteWorkspace to
inspect the active mirror root and files root before cleaning mirror data.

request_timeout_ms bounds ordinary sidecar/agent work and is the whole
install/update deadline in local-binary mode.
remote_agent_registry_timeout_ms is the whole automatic or explicit
install/update deadline in registry mode. It starts when the sidecar accepts
the request and covers host detection, retrieval, queue drain, staging, probes,
activation, rollback, and cleanup. Nested work receives only the remaining
budget, with a bounded reserve for recovery. The Neovim bootstrap callback adds
one second for reply delivery.
Install/update holds a per-target remote lease through its health recheck,
staging, activation, rollback, and cleanup. Concurrent sidecars either observe
the newly compatible agent or report install_in_progress; ambiguous stale
lease state fails closed. On POSIX, per-process claims protect the lease-owner
and operation-owner publication windows. Ownerless state is reaped only after
every well-formed claim identity is proven dead. Malformed claim names or file
types fail closed; dead partial contents are reaped by strict token/PID name.
Before upload, the installer publishes a stable same-directory transaction
journal. The next lease holder recovers it before probing: it removes only an
exact partial candidate, or restores the prior executable/finishes committed
cleanup only when the recorded paths, file state, and that journal's own
digests agree. A newer verified digest authorizes only its subsequent
transaction. Malformed, ambiguous, or file-hash-mismatched journals are
preserved and fail closed. POSIX install directories must be login-UID-owned,
non-symlink, and not group/world writable; new components use mode 0700 and
recovery journals use mode 0600.
Timeout, registry policy, and the automatic install path are fixed for the
lifetime of a connection. After replacing the configuration with setup(),
reconnect before automatic bootstrap uses those settings. Explicit command
arguments and defaults are evaluated when the command is invoked.
When a callback times out, the plugin sends a best-effort sidecar cancel for
queued remote work. Automatic agent bootstrap is the exception: once accepted,
its bounded transaction is not client-cancelable. Disconnect detaches Neovim
immediately but lets a stdio sidecar live through the snapped bootstrap callback
deadline, while a detached socket daemon can finish after its client channel
closes. Matching active read-only/background work is preempted by restarting the
serial SSH/agent worker. Newer interactive remote requests can also supersede
active read-only/background work; preempted results are normal no-op client
responses. Active save/flush work is not interrupted by that cancel.
Remote reads and writes use separate sidecar lanes with separate agent sessions;
unrelated opens, grep, validation, and probes can continue while save/flush work
is in flight, while reads that could conflict with pending writes stay ordered
on the write lane. Read/background lane backoff does not immediately block
write-lane save replay, and status still reports the latest lane error.
background_mirror controls the idle mirror builder; set it false and use
:RemoteMirrorStart manually on especially fragile links. Each background tick
scans metadata, hydrates a tiny clean-file batch, then validates a tiny
oldest-check cached-file batch against the remote checksum. The metadata scan
cursor is persisted across reconnects. Once a resumable scan reaches the end,
background_mirror_rescan_interval_ms prevents immediate full-tree rewalks.
grep_remote_page_files bounds each remote grep page by scanned files.
grep_remote_max_file_bytes skips oversized remote files without reading their
contents, and grep_remote_max_total_bytes stops a page before large no-hit
searches can monopolize the SSH worker. Consecutive remote pages use a live
agent continuation session when possible, and keep a path cursor as a restart
fallback.
recover_local_edits_on_connect scans hydrated local mirror files after reconnect
and queues snapshots for changed bytes that were saved while the sidecar was
unavailable, including crash-surviving saves that were never replayed from Lua
memory. recover_local_edits_limit bounds each local scan page.
Compare-and-swap conflicts keep the local save snapshot preserved as local
truth. Small remote conflict copies are stored completely; large remote conflict
copies are stored as a protocol-bounded prefix and reported with truncation
metadata. Accepting a remote conflict copy is only allowed for full saved remote
copies.
open_batch_max_file_bytes controls how large a direct uncached open may be
before it falls back from one-request batch read hydration to chunked reads.
auto_hydrate_mirror_buffers makes LSP jumps into missing local mirror files call
the sidecar open path and hydrate the corresponding remote file. Pending or
failed hydrations are read-only and are not flushed as remote saves.
adoption_policy defaults to "tracked_or_explicit", so new mirror-root files need
:RemoteAdopt before they are queued for remote creation. Set it to "auto" only
when mirror-root writes should be adopted as remote files automatically.

remote_runtime.enabled gates explicit process and terminal requests; it does
not execute a command during connect. remote_runtime.trust is "prompt",
"always", or "never". An accepted prompt grant is persisted per authority
in private local sidecar state. remote_runtime.ticket_create_timeout_ms bounds
local state preparation and private single-use ticket creation.
remote_runtime.detached_ttl_ms is reserved for the future persistent broker;
detached sessions are not available yet.
On Windows runtime state fails closed if an existing state_dir ancestor grants
an untrusted principal write-data, write-attributes, delete, child-delete,
DACL, or owner mutation. Use a pre-provisioned protected directory with
inheritance disabled and access limited to the current user and SYSTEM, or
repair the unsafe ancestor. Production does not create a drive-root directory
to bypass this check.
Private state limits cross-principal exposure; it is not a sandbox against a
process already running as the same account, or a Windows handle opened before
an ACL was tightened.

Agent and LSP launch commands share a sidecar transport planner. Local targets
spawn binaries directly; SSH targets use configured connection options and a
single shell-quoted remote command. Agent frames pass through an AgentSession
boundary and active work is interrupted through an abort-handle boundary. Stdio-
backed local/SSH child processes are the current session and abort
implementation. Future non-SSH sessions must provide reliable ordered delivery
per lane, matching request IDs, today's serial request semantics unless
multiplexing is added, backoff-compatible errors, and an abort path equivalent
to restarting the current lane worker. Future non-process transports should add
a session factory that returns the AgentSession and AgentAbortHandle pair while
preserving that lane-worker abort scope. Neovim still talks only to the sidecar
JSON API, so SSH stdio can be replaced later without changing editor-facing
behavior.
An SSH agent stream starts with one bounded stdout launch record before binary
RPC. Only an exact pre-exec FAILURE record may classify a missing or
non-executable agent for automatic repair; READY commits the stream to ordinary
agent handling. Child stderr and malformed, missing, oversized, or post-READY
records are diagnostic/untyped and cannot authorize replacement.
Sidecar command responses are newline-delimited JSON objects with an id, ok,
and result or error. The sidecar may also send id-less notification lines such
as workspace/remote_health; Neovim consumes those to keep last-known remote
health current, and future clients can ignore them without affecting command
responses. Runtime-v2 health is atomic: every active response/notification must
carry its complete runtime object. Malformed or conflicting equal-revision
objects are rejected, and lower revisions in the same workspace epoch are
ignored without partially updating legacy health fields.
The workspace_info command returns workspace identity, mirror paths, transport
metadata, supported command names, notification names, capabilities, and
last-known remote health from local state without starting or probing SSH.
Transport metadata includes generic kind, endpoint, connect_timeout_ms, and
agent_io fields while preserving SSH aliases. The legacy commands list includes
every implemented method for compatibility; new clients should check
capabilities.command_metadata, then use public_commands and command_specs to
distinguish public API from internal helpers such as flush_queued and to
identify local, remote, hybrid, or control commands. capabilities.agent_abort_scope
is currently lane_worker: preemption and shutdown abort and replace the active
serial worker for that lane, ignore late per-request replies from the replaced
worker, and reset handshake state through the same lifecycle as an SSH worker
restart. If the capability is
absent, fall back to commands. local means no remote dependency, not read-only.
command_specs also reports remote lane, mutation, fast-path, and preemption
metadata; clients should ignore unknown fields. hello is kept as a compatibility
alias for the same payload.
Workspace state keys preserve the legacy identity material: local for local
workspaces and the bare SSH target string for SSH workspaces. This keeps
existing mirrors stable, including the historical collision with an SSH target
named local. Future non-SSH transports should use namespaced identity material.

WORKSPACE RUNTIME                              *nvim-remote-mirror-runtime*

Workspace Runtime Readiness API v2 is the breaking provider-neutral contract
for plugins that need to run tools beside the authoritative tree. API v2
separates static provider support, dynamic authority readiness, and explicit
workspace authorization. The v2 support, readiness, preparation, facade, and
callback guarantees are one strict contract; there is no API-v1 compatibility
layer or fallback. Resolve an immutable context through the broker: >
  local runtime = require("nvim_remote_mirror.workspace_runtime")
  local context, err = runtime.resolve({ authority = "auto", bufnr = 0 })
<

Migration is breaking: replace the old context:authorize() plus direct-call
pattern with context:prepare() and methods on the returned prepared facade.
Never retry through legacy authorization or capability assumptions after a v2
readiness, trust, or stale-state error. API-v2 direct calls remain defined by
the strict state matrix below; they are not an API-v1 adapter.

The query accepts authority="auto"|"local"|"remote" and at most one of bufnr,
path, workspace_id, or workspace_key. Auto precedence is remote buffer
ownership, current-tab remote binding, then local. Successful connect binds
its originating tab. Disconnect retains an offline binding and never falls
back local; :RemoteUseLocal clears only that tab binding.
require("nvim_remote_mirror").workspace() remains the remote-only compatibility
resolver with its active-workspace fallback. Public context fields include
api_version, provider, workspace_id, epoch, state, mode, authority, and roots.
Public context.capabilities was removed; query support and readiness through
methods instead.
An offline-buffer context reports supported capabilities as unchecked (or
disabled by policy), but prepare() and execution fail with workspace_offline.

Contexts are tied to a reconnect epoch. Readiness has a separate monotonic
revision. context:is_current() checks a retained context; reconnect or
transport replacement returns stale_context from a context, while an epoch or
readiness change returns stale_preparation from a prepared facade. Resolve and
prepare again rather than retrying stale state.

context:map_path(path, { from=..., to=... }) crosses between editor,
authority, editor_uri, and authority_uri spaces. It rejects paths outside the
workspace, invalid encodings, controls, UNC paths, and invalid Windows roots.
For example: >
  local authority_uri = assert(context:map_path(
    vim.api.nvim_buf_get_name(0),
    { from = "editor", to = "authority_uri" }
  ))
<

Use supports() for static local support, capability_status() for the latest
local readiness snapshot, and prepare() for execution: >
  if not context:supports("process") then
    return
  end
  local status = assert(context:capability_status("process"))
  -- state: unsupported, disabled, unchecked, checking, ready, or unavailable.

  context:prepare("process", function(prepare_err, prepared)
    if prepare_err then
      return
    end
    local handle = assert(prepared:spawn({
      command = { argv = { "cargo", "check" } },
      cwd = { space = "workspace", path = "" },
    }))
  end)
<

For every well-formed accepted prepare call, its callback runs exactly once.
It may run inline when readiness and trust are cached, or later after a probe
or prompt. Initialize callback-visible state before invoking prepare(). Invalid
arguments return nil,error synchronously and do not invoke the callback.
Preparation performs a read-only agent probe, verifies the negotiated
capability, applies trust policy, and rechecks epoch and readiness revision. It
may persist an explicit local trust decision, but never installs or updates the
remote agent.

Automatic signed repair remains connect-only and requires SSH, a trusted
configured registry, and remote_agent_auto_install=true. Missing or
incompatible agents otherwise produce a typed readiness error; explicit
:RemoteInstallAgent and :RemoteUpdateAgent remain separate actions.

The prepared facade is the recommended revision-bound receiver. A process
facade can start only pipe stdio and a terminal facade only PTY stdio; a
mismatch returns unsupported, and a later epoch/readiness change returns
stale_preparation.

Direct context:job_spec(), context:spawn(), and context:open_pty() are retained
as a v2 late-check path. They allow unchecked or ready-and-effective state
after existing trust. Checking or unavailable authority returns
capability_not_ready; disabled, unsupported, and explicitly unnegotiated
capabilities return capability_disabled, unsupported, and
capability_unavailable. The proxy always repeats the definitive Hello at
launch.

Process specs keep command and environment structured: >
  local process = {
    command = { argv = { "cargo", "check" } },
    cwd = { space = "workspace", path = "" },
    env = { clear = false, set = { NO_COLOR = "1" }, unset = { "PAGER" } },
    stdio = "pipe",
    persistence = "attached",
    timeout_ms = 30000,
    max_output_bytes = 4 * 1024 * 1024,
  }
<

command may instead be { shell = "default" }. Arbitrary shell text is not
accepted there; use explicit argv such as { "sh", "-lc", "make test" } when
shell evaluation is intentional. cwd space may be workspace-relative,
buffer-relative, or an absolute contained editor/authority path.

Arguments are UTF-8 and control-free on every platform. max_output_bytes is a
pipe-only cumulative limit (4 MiB by default, 128 MiB maximum). Omit it for
PTYs: PTY output is a live backpressured stream with no cumulative byte limit,
and a PTY request containing max_output_bytes is rejected.

Use prepared:job_spec(process) when another plugin owns the local job. It
returns authoritative bridge argv and a canonical command string for
string-only APIs. Broker results add public authority identity and
input.newline. Remote argv, cwd, and environment stay inside a private,
single-use ticket; the local bridge contains only nrm-sidecar and an opaque
ticket ID. Create a new job_spec for every start and never append shell text
containing remote or user-provided values to its command.

The optional ToggleTerm adapter manages that generic bridge: >
  require("nvim_remote_mirror.integrations.toggleterm").toggle({
    key = "shell",
    direction = "float",
    query = { authority = "auto" },
  })
<

ToggleTerm's own setup() must complete before this adapter is called. A lazy
command stub is rejected with toggleterm_not_initialized and is not replaced.
The supported and CI-tested compatibility baseline is ToggleTerm v2.13.1. The
adapter passes the authority newline ("\r" for Windows PowerShell),
coalesces pending opens, reuses a hidden live PTY, and creates a new prepared
facade, ticket, and Terminal after exit. It never falls back to a local shell.
Broker-owned terminals are hidden with IDs outside Neovim's Ex-count range,
and the adapter replaces raw :ToggleTerm with a local-only path. Advanced
upstream commands that explicitly include hidden terminals or a private
managed ID are outside that escape-hatch guarantee. Public third-party
workspace-provider registration is not exposed by this contract.

Use prepared:spawn(process, handlers) for a managed attached pipe or PTY, or
prepared:open_pty(process, handlers) for a prepared terminal. Handles expose
write(), close_stdin(), signal(), and kill(); PTYs also expose resize().
Signals are interrupt, terminate, kill, or hangup. on_stdout/on_stderr receive
Neovim line/list job stream callbacks, not a binary-transparent byte API.
Embedded NUL bytes are represented as line breaks by Neovim; use job_spec with
an argv-capable raw process consumer when exact bytes are required. on_exit
receives the usual id/code/event plus a
fourth structured authority result when it is available.

The local managed provider enforces timeout and cumulative pipe-output limits
and applies PTY character dimensions. Local PTY pixel dimensions are
unsupported and fail before spawn. A job_spec consumer owns its bridge
lifecycle and must apply limits supported by its process API.

User NrmWorkspaceConnected, NrmWorkspaceDisconnected, and
NrmWorkspaceEpochChanged events contain epoch, workspace_key, target, and
state plus a reason/reconnect field when applicable. Treat event data as a
hint, discard cached contexts, and resolve again before execution.

User NrmWorkspaceReadinessChanged contains the normal epoch, workspace_key,
target, and connection state fields plus readiness_state, readiness_revision,
and effective. It does not advance the workspace epoch. Retain the context if
desired, discard prepared facades, call capability_status(), and prepare again.

The Lua open_terminal(opts, callback) helper is callback-based in API v2.
Every accepted call invokes callback(err, handle) exactly once. Cached
readiness and trust may complete inline; otherwise a probe or prompt completes
later. Initialize callback-visible state before invoking it. :RemoteTerminal
reports failures through notification regardless of callback timing.

Only attached pipes and PTYs are advertised today. Detached/reconnectable
sessions and workspace_watch_v1 are unavailable. [!] on :RemoteTerminal
therefore returns persistence_unavailable. Runtime execution is orchestration,
not a sandbox: commands run as the authority account, and a deliberately
daemonized POSIX descendant may outlive attached signal handling.

See docs/workspace-runtime.md in the project checkout for the complete API,
trust/ACL policy, limits, error codes, and integration guidance.

PICKER API                                      *nvim-remote-mirror-pickers*

The generic picker module uses sidecar-backed remote file and grep results with
builtin vim.ui.select selection. Plugin-specific Telescope/fzf/snacks sources
are not implemented yet; unknown or unavailable providers warn and use builtin
selection.
>
  local pickers = require("nvim_remote_mirror.pickers")

  pickers.files({
    query = "src",
    on_select = function(item)
      require("nvim_remote_mirror").open(item.path)
    end,
  })

  pickers.grep({ query = "TODO" })

  require("nvim_remote_mirror").grep_async("TODO", { limit = 50 }, function(err, result)
    if not err then
      vim.print(result.hits)
    end
  end)
<

SIGNED AGENT REGISTRY                       *nvim-remote-mirror-registry*

The registry is opt-in. remote_agent_registry_url must use https:// or a local
absolute file:// URL and contain exactly one {version} placeholder. Configure
trusted 32-byte Ed25519 public keys with out-of-band key IDs; the default
signature threshold is one. Registry responses can never add trusted keys.

To use the project's signed GitHub releases: >
  require("nvim_remote_mirror").setup({
    remote_agent_auto_install = true,
    remote_agent_registry_url =
      "https://github.com/jkammerland/nvim-remote-mirror/releases/download/v{version}/nrm-agent-manifest-v1.json",
    remote_agent_registry_public_keys = {
      ["release-2026-q3"] = "o70a35HCIieJ/B0jatGVvNB/6l3X2W4InbioQjIFHbY=",
    },
  })
<

Automatic or explicit install/update detects Linux, macOS, or Windows
x64/ARM64, verifies detached signatures and the strict manifest locally,
rechecks the selected artifact size and SHA-256, and then uploads it through
SSH. Registry mode fails closed: it never falls back to the configured local
agent. Automatic install requires SSH, this trusted registry, and
remote_agent_auto_install=true; no registry is configured by default. It
repairs only missing_agent, agent_not_executable, version_mismatch, or
protocol_mismatch. Other health states are not mutated.

The default 512 MiB cache lives below sidecar state unless
remote_agent_registry_cache_dir is set. Current keys, signature threshold,
manifest policy, artifact size, and digest are reverified on every use. A cached
manifest/signature pair is eligible only after timeout, connection, rate-limit,
or 5xx failures; malformed data, invalid signatures or hashes, other 4xx
responses, and policy violations never use that fallback. Content-addressed
artifacts can be normal cache hits after fresh manifest verification. Budget
eviction means offline fallback is best-effort, not guaranteed.

Install/update results and the Bootstrap/Registry sections of :RemoteWorkspace
include automatic state, result/reason/error, selected platform/target,
redacted manifest URL, verified key IDs, digests, source/cache state, and stable
error codes. A failed automatic bootstrap leaves connect degraded but complete.
Registry state remains separate from working-agent health.
An automatic result is considered ready only when its remote health contains a
valid ready runtime-v2 observation in addition to exact package/protocol
compatibility. That observation seeds plugin readiness immediately; missing or
malformed runtime health leaves bootstrap in the degraded error state.

See docs/agent-registry.md in the project checkout for the exact manifest,
redirect, cache, trust, and key-rotation policy.

==============================================================================
LIMITS                                        *nvim-remote-mirror-limits*

This is dogfooding software. The local mirror, SSH workspaces, background
metadata scan, find, grep, save queue, conflicts, socket sidecar mode, and basic
LSP proxy are implemented. Remote git status/diff/blame primitives are
implemented. Attached remote pipe processes and PTYs are available; the strict
Workspace Runtime Readiness API-v2 migration is work in progress.
:RemoteTerminal uses the callback-based readiness path. Persistent/detached
terminals,
workspace watching, DAP, full git adapters, and general plugin remoting are
not built yet.

Local plugins operate on hydrated mirror files. Local grep or file pickers can
only see what is already known or present in the mirror; use :RemoteScan,
:RemoteFind, :RemoteFiles, and :RemoteGrep for sidecar-backed discovery.

Saving a new file under the mirror files root does not create it remotely by
default. Use :RemoteAdopt {path} for files that should exist remotely, or set
adoption_policy = 'auto' if new mirror-root files should be adopted on write.
After :RemoteCd, cwd-based plugins may write generated files under the mirror
root. Until ignore policy is configurable, keep plugin caches, temp files, tags,
and build outputs outside the mirror root unless they should exist remotely.

==============================================================================
LSP                                                *nvim-remote-mirror-lsp*

After :RemoteConnect completes, use start_lsp() or :RemoteLspStart to run a
language server on the remote side while Neovim receives local mirror paths.
The proxy starts the server in the remote workspace directory. SSH targets use
the configured connect timeout and cd to the remote root before server launch.
start_lsp() probes the remote first and skips vim.lsp.start() while the remote
is unavailable or in backoff, recording the last LSP error for :RemoteLspStatus.
lsp_client_config() remains available as a low-level config builder when you
need manual control.
Started LSP client IDs are tracked by workspace identity. :RemoteDisconnect and
:RemoteLspStop stop only clients for the matching remote workspace; stale async
starts are ignored when the active workspace changes.
The proxy rewrites file:// URI values, workspace-edit URI object keys, and known
path fields between local mirror and remote workspace prefixes. Plain path
prose strings are left untouched and path boundaries are checked before
rewriting. Windows remote prefixes accept B:/..., B:\..., and file:///B:/...
forms and emit canonical drive paths/URIs while PowerShell relays preserve raw
LSP stdio bytes. Native executables and .cmd/.bat shims resolve from PATH.
Batch arguments containing " or % are rejected rather than passed through
unsafe cmd.exe expansion; use a native executable or fixed wrapper instead.

>
  local nrm = require("nvim_remote_mirror")
  nrm.start_lsp({ "rust-analyzer" }, {
    name = "remote-rust-analyzer",
  })

  nrm.start_lsp({
    name = "remote-rust-analyzer",
    cmd = { "rust-analyzer" },
  })
<
